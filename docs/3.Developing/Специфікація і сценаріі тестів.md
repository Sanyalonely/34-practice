## **1. ВХІДНІ СПЕЦИФІКАЦІЇ**
Цей розділ регламентує всі вхідні дані, події та системні запити, необхідні для успішного проходження тестування веб-сайту "NotatkaHUB".

## 1.1. Дані користувача (Форми вводу)

| № | Найменування входу | Характеристики (Тип даних) | Значення / Приклад      |  Інтерфейс    |
|---|--------------------|----------------------------|-------------------------|---------------|
| 1 | **Логін**          | String (Alphanumeric)      | `Yarik_Tester`, `admin` | Текстове поле |
| 2 | **Пароль**         | String (Masked)            | `Qwerty12345!`          | Поле Password |
| 3 | **Заголовок**      | String (Max 100 chars)     | `План на день`          | Input field   |
| 4 | **Текст нотатки**  | Text (Unicode support)     | Довільний текст, емодзі | Textarea      |
| 5 | **Пошуковий запит**| String                     | `Проект`, `2026`        | Search bar    |

## 1.2. Дії користувача (Події інтерфейсу)

| № | Назва дії | Опис входу | Характеристики | Тип пристрою |
|---|-----------|------------|----------------|--------------|
| 1 | **Клік (Click)** | Натискання кнопок «Зберегти», «Видалити» | Mouse Left Click | Миша / Тачпад |
| 2 | **Навігація** | Перехід за URL-адресою | `GET /notes/edit/1` | Адресний рядок |
| 3 | **Скролінг** | Прокрутка списку нотаток | Wheel/Touch event | Миша / Екран |
| 4 | **Hotkeys** | Комбінації клавіш (напр. `Ctrl+S`) | Keyboard Event | Клавіатура |

## 1.3. Системні входи та протоколи

| № | Найменування | Характеристики | Протокол / Місце |
|---|--------------|----------------|------------------|
| 1 | **Auth Token** | JWT (JSON Web Token) | HTTP Header (Authorization) |
| 2 | **Cookies** | Session ID | Browser Storage |
| 3 | **JSON Payload** | Структура `{ "title": "...", "body": "..." }` | HTTP POST Request |
| 4 | **API Responses** | Коди відповідей (200, 404, 500) | REST API |

## **2. ВИХІДНІ СПЕЦИФІКАЦІЇ**
Цей розділ визначає очікувані результати роботи системи, які підлягають перевірці під час тестування веб-сайту "NotatkaHUB".

## 2.1. Візуальні та інформаційні виходи (UI)

| № | Найменування виходу | Характеристики (Тип) | Опис результату |
|---|---------------------|----------------------|-----------------|
| 1 | **Список нотаток** | Елементи інтерфейсу  | Відображення карток із заголовком та датою створення. |
| 2 | **Статус-повідомлення** | String (Toast/Alert) | Текст: "Нотатку успішно збережено", "Помилка авторизації". |
| 3 | **Пошукова видача** | Фільтрований масив   | Відображення лише тих нотаток, що відповідають запиту. |
| 4 | **Повідомлення про валідацію** | Текст червоного кольору | "Пароль надто короткий", "Поле не може бути порожнім". |

## 2.2. Системні та технічні виходи

| № | Найменування | Тип даних / Формат | Опис |
|---|--------------|--------------------|------|
| 1 | **HTTP Status Codes** | Integer | `200 OK` (успіх), `201 Created`, `403 Forbidden`, `500 Error`. |
| 2 | **API Response** | JSON Object | Структура: `{ "id": 101, "status": "success", "timestamp": "..." }`. |
| 3 | **Записи в БД** | Row in SQL Table | Створення нового рядка в таблиці `Notes` з прив'язкою до `UserID`. |
| 4 | **Логи сервера** | .log file / Stream | Записи про час виконання запитів та виникнення винятків (Exceptions). |

## 2.3. Результати тестування (Artifacts)

Як результат проходження тестів, тестувальник очікує отримати:

* **Звіти про швидкодію:** Звіт від **Google Lighthouse** у форматі PDF/JSON (показники LCP, TTI).
* **Скріншоти (Screenshots):** Знімки екрана з багами або некоректним відображенням у різних браузерах (через **BrowserStack**).
* **Трафік-дампи:** Логи перехоплених запитів у **Fiddler** для перевірки структури API-пакетів.
* **Тест-репорти:** Автоматично згенеровані звіти про успішне/неуспішне проходження Unit-тестів (**NUnit/JUnit**).

## **3. СЕРЕДОВИЩЕ ТЕСТУВАННЯ**
Цей розділ описує технічні характеристики та конфігурацію оточення, необхідного для проведення повноцінного тестування веб-сайту «Нотатник».

## 3.1. Програмне забезпечення (Software)

Для виконання тестів використовується наступний стек системного та прикладного ПЗ:

| Категорія | Найменування | Характеристики |
|---|---|---|
| **Операційні системи** | Windows 10/11, macOS Sonoma, Android 13+, iOS 17 | Основні платформи для перевірки десктопної та мобільної версій. |
| **Браузери (Desktop)** | Google Chrome (LTS), Mozilla Firefox, Microsoft Edge, Safari | Останні стабільні версії для кросбраузерного тестування. |
| **Браузери (Mobile)** | Chrome Mobile, Safari Mobile | Перевірка адаптивності інтерфейсу. |
| **Середовища розробки** | IntelliJ IDEA / Visual Studio Code | Для запуску та налагодження юніт-тестів. |
| **Веб-сервер** | Apache / Nginx / Node.js Runtime | Локальне середовище для запуску бекенд-частини нотатника. |

## 3.2. Апаратне забезпечення (Hardware)

Мінімальні та рекомендовані характеристики обладнання для робочого місця тестувальника:

* **Процесор:** Intel Core i5 / AMD Ryzen 5 (або еквівалент) і вище.
* **Оперативна пам'ять:** Мінімум 16 ГБ (рекомендовано 16 ГБ для стабільної роботи Docker та декількох браузерів одночасно).
* **Мережеве обладнання:** Стабільне інтернет-з'єднання (мін. 10 Мбіт/с) для взаємодії з хмарними сервісами (BrowserStack, Loadium).
* **Мобільні пристрої:** Наявність хоча б одного фізичного смартфона (iOS/Android) для фінальної перевірки UX.

## 3.3. Спеціалізоване ПЗ та налаштування

Для виконання специфічних видів тестів розгорнуто наступні інструменти:

*  **BrowserStack:** Хмарна інфраструктура для доступу до 3000+ комбінацій реальних пристроїв та браузерів.
*  **Fiddler Everywhere:** Налаштований як проксі-сервер для перехоплення та дешифрування HTTPS-трафіку між браузером та API нотатника.
*  **Google Lighthouse (Chrome DevTools):** Вбудований інструмент для автоматизованого аудиту продуктивності.
*  **JUnit / NUnit Frameworks:** Інтегровані в CI/CD конвеєр для автоматичного запуску тестів при кожному Commit.

## 3.4. Конфігурація середовища

* **Тестова база даних:** Окремий екземпляр БД (напр. PostgreSQL/MySQL), заповнений тестовими наборами даних (dummy data).
* **Доступ:** Для тестування безпеки створено декілька облікових записів із різними ролями (Admin, User, Guest).
* **CORS Settings:** Налаштовані дозволи для взаємодії фронтенду з тестовим API сервером.
