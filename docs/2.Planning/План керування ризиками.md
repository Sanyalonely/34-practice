## 1. СПИСОК ГОЛОВНИХ РИЗИКІВ

### 1.1. Вступ
На етапі планування ризиків, грунтуючись на попередній оцінці, ми виділили п'ять найбільш критичних чинників. Для цих ризиків розроблено детальні стратегії запобігання та плани реагування, щоб мінімізувати їхній вплив на успіх проекту NotatkaHUB.

### 1.2. Таблиця головних ризиків

| Пріоритет | Причина (Ризик) | Наслідок | Ймовірність | Загроза | Очікувана величина |
| :--- | :--- | :--- | :---: | :---: | :---: |
| 1 | Помилки API | Нотатки не зберігаються | 50% | 5 | 2.5 |
| 2 | Сервер недоступний | Користувач не може отримати доступ до сервісу | 40% | 5 | 2.0 |
| 3 | Помилка підключення до MongoDB | Дані не записуються у базу | 40% | 5 | 2.0 |
| 4 | Проблеми адаптивності UI | Поганий UX на мобільних пристроях | 40% | 3 | 1.2 |
| 5 | Помилки MongoDB запитів | Пошкодження або некоректна вибірка даних | 30% | 4 | 1.2 |

## 2. ПЛАНУВАННЯ ЗАХОДІВ

### 2.1. Вступ
Для роботи з ризиками ми застосовуємо комбінований підхід, що включає дослідження складних ділянок, запобігання виникненню помилок та підготовку планів на випадок настання негативних подій.

### 2.2. Аналіз підходів для головних ризиків

**Ризик 1: Помилки API**
- **Дослідження:** Вивчення стандартів REST та інструментів валідації даних.
- **Погодження:** Неприпустимо, оскільки це основна функціональність.
- **Уникнення:** Використання TypeScript для суворої типізації.
- **Перенесення:** Неможливе.
- **Запобігання:** Покриття ендпоінтів Unit-тестами та автоматизація тестування в CI/CD.
- **Пом'якшення наслідків:** Впровадження системи логування (Sentry) для швидкої ідентифікації та виправлення помилок.

**Ризик 4: Проблеми адаптивності UI**
- **Дослідження:** Аналіз статистики популярних розширень екранів серед цільової аудиторії.
- **Погодження:** Можливо для рідкісних розширень (напр. надмалі екрани).
- **Уникнення:** Використання сучасних CSS-фреймворків з Mobile-first підходом.
- **Перенесення:** Використання готових бібліотек компонентів.
- **Запобігання:** Регулярне тестування інтерфейсу в Chrome DevTools на різних пресетах.
- **Пом'якшення наслідків:** Спрощення інтерфейсу (fallback до базового списку) при виявленні помилок верстки.

## 3. ДОСЛІДЖЕННЯ
Для мінімізації технологічної невизначеності ми проведемо наступні дослідження:
* **Ризик "Помилки MongoDB запитів":** Дослідити документацію Mongoose щодо роботи з транзакціями та індексами для уникнення race conditions.
* **Ризик "Сервер недоступний":** Проаналізувати SLA провайдерів для вибору найбільш стабільної платформи.

## 4. ПОГОДЖЕННЯ
Ми погоджуємося з наступними ризиками:
* **Ризик "Повільні запити до MongoDB":** На етапі MVP затримка до 300 мс вважається прийнятною, оскільки обсяг даних користувачів буде невеликим. Спеціальні заходи з шардингу на цьому етапі не впроваджуються.

## 5. УНИКНЕННЯ
* **Ризик "Втрата даних у БД":** Уникаємо ризику шляхом використання хмарного сервісу MongoDB Atlas, який автоматично керує реплікацією та збереженням даних на рівні інфраструктури.

## 6. ПЕРЕНЕСЕННЯ
* **Ризик "Сервер недоступний":** Переносимо відповідальність за доступність серверної частини на PaaS-провайдера (наприклад, Vercel), що гарантує високий рівень Uptime без необхідності власного адміністрування.

## 7. ЗАПОБІГАННЯ
| Ризик | Захід запобігання | Оцінка витрат часу |
| :--- | :--- | :--- |
| Помилки API | Написання автотестів (Jest) | 20 годин |

| Проблеми адаптивності UI | Використання адаптивної сітки (Grid/Flexbox) | В межах розробки |

## 8. ПОМ'ЯКШЕННЯ НАСЛІДКІВ
| Ризик | План реагування | Тригер |
| :--- | :--- | :--- |
| Помилки API | Відкат до попередньої стабільної версії (Rollback) | Ріст помилок 5xx > 5% |
| Сервер недоступний | Перемикання на резервний регіон/провайдер | Downtime > 15 хв |
| Помилки запитів DB | Відновлення даних з щоденного бекапу | Виявлення пошкоджених записів |

## 9. ПЛАНУВАННЯ КЕРУВАННЯ РИЗИКАМИ

### 9.1. План для Ризику №1: Помилки API
* **Ідентифікатор:** R01
* **Формулювання:** Через неправильну логіку обробки запитів на сервері, нотатки користувача можуть не зберігатися, що призведе до втрати контенту.
* **Стратегія запобігання:** Впровадження обов'язкового модульного тестування та валідації схем даних.
* **Плановані дії:** 1. Розробка схем валідації (до 15.03) - Відповідальний: Пилипенко О.О.
  2. Налаштування CI-пайплайну (до 20.03) - Відповідальний: Абоімов В.П.
* **Тригер:** Помилка валідації на сервері або HTTP 500.
* **Відповідальний:** Пилипенко О.О.

### 9.2. План для Ризику №3: Помилка підключення до MongoDB
* **Ідентифікатор:** R03
* **Формулювання:** Неправильна конфігурація або збій мережі призводить до неможливості з'єднання з базою даних.
* **Стратегія запобігання:** Використання змінних оточення (ENV) та моніторинг стану підключення при старті сервера.
* **Плановані дії:**
  1. Реалізація механізму повторного підключення (Retry) (до 12.03) - Відповідальний: Абоімов В.П.
  2. Налаштування сповіщень у Telegram про падіння БД (до 18.03) - Відповідальний: Ленко А.Е.
* **Тригер:** Подія `mongoose.connection.on('error')`.
* **Відповідальний:** Абоімов В.П.
