## 2. ПЛАН РОЗРОБКИ

Цей розділ містить огляд основних складових плану розробки проекту **"NotatkaHUB"**. Наведено цілі, стратегію, часові межі та технологічний стек, що використовуватиметься командою "СВАРЩИКИ".

---

### 2.1. Огляд об'єкта розробки
Згідно з концепцією рішення (п.2.5), буде розроблено клієнт-серверний веб-застосунок, що включає:
* **Модуль автентифікації:** Реєстрація та вхід користувачів для доступу до персональних нотаток.
* **Markdown-редактор:** Повнофункціональне поле для введення тексту з підтримкою синтаксису Markdown.
* **Система управління нотатками:** Створення, редагування, видалення та перегляд (CRUD).
* **Чернетка для нотаток:** При виході зі сторінки створення нотатки, заповнені поля збережуться та при наступному створенні, будуть мати ті самі значення.
* **Експорт та імпорт:** Можливість завантажити собі на пристрій нотатки у форматі `.md`, також є можливість імпортувати у редактор нотатку.
* **Інтерфейс:** Адаптивна верстка з підтримкою світлої та темної тем.

### 2.2. Модель розробки
Для реалізації проекту обрано **Водоспадну модель (Waterfall)**. Процес поділено на послідовні етапи, де завершення одного є сигналом до початку наступного.

    2.  **Розробка (Implementation):** (тиждень 3) — написання коду фронтенду та бекенду.
    3.  **Тестування та стабілізація:** (тиждень 4) — виправлення багів.
    4.  **Розгортання:** (фінальні 3 дні) — деплой на Vercel та налаштування MongoDB Atlas.
       
* **Період активної розробки:** з 13 березня 2026 р. по 22 березня 2026 р.
  
* **Календарний план написання коду:**

    2.  **Розробка** (16 – 20 березня) — основна робота над функціоналом фронтенду та бекенду.
    3.  **Корегування та перевірка** (20 – 22 березня) — рефакторинг коду, перевірка згідно з дизайном та вимогами.

### 2.3. Технологічний стек
Для створення рішення використовуються наступні інструменти та технології (згідно з п.3.2 Концепції):

### Frontend (Клієнтська частина)
* **Framework:** React (Vite)
* **Language:** TypeScript
* **Styles:** Tailwind та Css modules
* **State Manager** Zustand (для керування глобальним станом)
* **Markdown Parsing:** `react-markdown`
* **HTTP Client:** Axios (для зв'язку з API)

#### Backend (Серверна частина)
* **Runtime:** Node.js
* **Framework:** Express.js
* **Auth:** JWT (JSON Web Tokens) & bcrypt (хешування)

#### Database & Infrastructure
* **DB:** MongoDB (Mongoose ODM)
* **Cloud DB:** MongoDB Atlas
* **Hosting:** Vercel (Frontend)
* **Version Control:** Git / GitHub

### **3.1. Стратегія тестування**
Стратегія базується на поступовому переході від перевірки окремих компонентів до повного тестування системи в реальних умовах:

* **Пріоритетність:** Спочатку тестуються критичні функції (збереження даних, авторизація), потім — допоміжні (пошук, дизайн).
* **Повторюваність:** Будь-яке виправлення багу супроводжується обов’язковим **регресійним тестуванням**.
* **Кросплатформеність:** Використання **BrowserStack** для перевірки ідентичного вигляду сайту на Windows, iOS та Android.

### **3.2 Особливості тестування**
Процес перевірки «Нотатника» має наступні специфічні нюанси:

* **Цілісність даних:** Перевірка, чи не зникають нотатки при раптовому закритті вкладки або розриві зв'язку.
* **Обробка тексту:** Тестування підтримки емодзі, спеціальних символів та великих обсягів тексту в одній нотатці.
* **Приватність:** Гарантія того, що користувач "А" не може побачити нотатки користувача "Б", навіть змінивши ID у посиланні.

   - Підготовка `Dockerfile` для серверної частини (Backend) для забезпечення ідентичності середовища розробки та виконання.

3. **Розгортання Backend**: 
   - Створення Web Service на платформі Render.
   - Підключення GitHub-репозиторію.
   - Налаштування Root Directory (src/backend) та налаштування змінних оточення (зокрема `Mongodb_URI`).

4. **Розгортання Frontend**: 
   - Створення проекту на Vercel.
   - Налаштування Root Directory (src/frontend) та встановлення зв'язку з бекендом через змінну оточення `Render_URI`.

5. **Налаштування CI/CD**: 
   - Активація автоматичного перезбирання проектів при кожному оновленні (push) у головну гілку GitHub-репозиторію.

### **5.2. Особливості оплати та використання ресурсів**

* **Модель ресурсів:** Проект базується на моделі **BYOD (Bring Your Own Device)**, де розробники використовують власну техніку. Це дозволяє уникнути значних капітальних інвестицій на старті.

* **Гнучкість:** У разі масштабування проекту, перехід на платні тарифи (наприклад, MongoDB Dedicated Cluster або GitHub Enterprise) потребуватиме перегляду бюджету в сторону збільшення на 15-20%.
