## **1. НЕОБХІДНІСТЬ ПРОЕКТУ**

### **1.1. Обгрунтування необхідності**

Проект спрямований на створення простого клієнтського веб-застосунку для ведення нотаток у форматі Markdown без необхідності встановлення додаткового програмного забезпечення.

На основі інтерв’ю із замовником визначено такі потреби:

- створення, редагування, перегляд та видалення нотаток;

- автоматичне збереження під час редагування;

- підтримка Markdown-форматування;

- можливість експорту нотаток у форматі .md;

- можливість експорту як окремої нотатки, так і всіх нотаток;

- адаптивний інтерфейс для роботи з різних пристроїв;

- перемикання між світлою та темною темами;

- просте перемикання між режимами редагування та перегляду (кнопка з іконкою олівця).

Додаткові можливості (теги, імпорт, фільтрація тощо) можуть бути реалізовані як розширення, але не є обов’язковими.

### **1.2. Бачення проекту**

Протягом одного місяця розробити клієнтський веб-застосунок для створення та збереження Markdown-нотаток з автозбереженням і експортом у форматі .md, який працює у браузері без встановлення додаткового ПЗ.

### **1.3. Аналіз вигод**

**Користувачі отримають:**

- швидке створення форматованих нотаток;

- автоматичне збереження без ризику втрати даних;

- зручне перемикання між редагуванням і переглядом;

- експорт нотаток у стандартному Markdown-форматі;

- можливість зберегти всі нотатки одним пакетом;

- комфортну роботу завдяки світлій і темній темам;

- доступність застосунку з будь-якого пристрою з браузером.

**Замовник отримає:**

- завершений сучасний веб-продукт;

- основу для подальшого розвитку (імпорт, теги, синхронізація);

- приклад реалізації клієнтської архітектури SPA.
 
## **2. КОНЦЕПЦІЯ РІШЕННЯ**

### **2.1. Цілі та Завдання**

**Мета 1: Реалізувати базову функціональність нотатника**

- створення нової нотатки;

- редагування існуючої нотатки;

- перегляд нотатки;

- видалення нотатки;

- автозбереження під час редагування.

**Мета 2: Забезпечити підтримку Markdown**

- інтеграція Markdown-парсера;

- відображення форматованого тексту;

- перемикання між режимами редагування та перегляду кнопкою (іконка олівця).

**Мета 3: Реалізувати експорт**

- експорт однієї нотатки у форматі .md;

- експорт усіх нотаток;

- коректне формування файлів для завантаження.

**Мета 4: Забезпечити зручний інтерфейс**

- адаптивна верстка (mobile-first);

- світла та темна тема;

- простий мінімалістичний інтерфейс;

- автоматичне збереження змін.

### **2.2. Припущення і Обмеження**

#### Припущення

- користувач працює в сучасному браузері;
  
- користувач має доступ до Інтернету для роботи із застосунком;
  
- автозбереження здійснюється при зміні вмісту;
  
- нотатки зберігаються у базі даних MongoDB;
  
- кожна нотатка прив’язана до конкретного користувача;
  
- користувач повинен зареєструватися або увійти в систему;
  
- теги, сортування, фільтрація не є обов’язковими;
  
- імпорт нотаток може бути реалізований як додаткова функція;
  
- достатньо двох тем (світла/темна).

#### Обмеження

- проект потребує серверної частини (backend);
  
- використовується база даних MongoDB для збереження нотаток;
  
- термін виконання – 1 місяць;
  
- drag-and-drop не реалізується;
  
- налаштування шрифтів та розміру тексту не входить у поточну версію;
  
- експорт здійснюється лише у форматі `.md`;
  
- система авторизації реалізується у базовому вигляді (реєстрація та вхід).

### **2.3. Аналіз використання**
**Групи користувачів**

- Студенти

- Викладачі

- Користувачі для особистих нотаток

**Основні сценарії використання**

- Створення нової нотатки

- Автоматичне збереження при редагуванні

- Перемикання між режимом редагування та перегляду

- Видалення нотатки

- Експорт окремої нотатки

- Експорт усіх нотаток

- Перемикання світлої/темної теми

### **2.4. Вимоги**
**Вимоги користувачів**

- CRUD-операції для нотаток;
  
- автозбереження;
  
- підтримка Markdown;
  
- кнопка перемикання режиму редагування/перегляду (іконка олівця);
  
- експорт однієї нотатки у `.md`;
  
- пакетний експорт усіх нотаток;
  
- адаптивний інтерфейс;
  
- світла та темна тема;
  
- реєстрація користувача;
  
- авторизація користувача;
 
- збереження нотаток у персональному акаунті.

**Системні вимоги**

- робота у сучасних браузерах;
  
- серверна частина для обробки запитів (backend);
  
- збереження даних у базі даних **MongoDB**;
  
- використання REST API для взаємодії клієнта і сервера;
  
- можливість розгортання клієнтської частини на **GitHub Vercel**, а серверної частини на **хмарному сервері**;
  
- мінімальні апаратні вимоги (будь-який пристрій із браузером та доступом до Інтернету).

### **2.5. Рамки**
**Функціональність рішення**

- створення, редагування, перегляд, видалення нотаток;

- автозбереження;

- Markdown-підтримка;

- експорт .md (однієї або всіх нотаток);

- адаптивність;

- світла/темна тема;

- перемикання режиму редагування кнопкою з іконкою олівця.

**За рамками рішення**

- теги;

- пошук по нотатках;

- сортування та фільтрація;

- функція «закріпити» нотатку;

- імпорт нотаток (можливе розширення);

- drag-and-drop;

- налаштування шрифтів;

- експорт у PDF або TXT;

- хмарна синхронізація.

### **2.6. Критерії схвалення рішення**

Рішення буде вважатися готовим, якщо:

- реалізовано повну обов’язкову функціональність;

- автозбереження працює стабільно;

- коректно експортуються .md-файли (одна та всі нотатки);

- реалізовано перемикання світлої/темної теми;

- застосунок адаптивний;

- відсутні критичні помилки;

- застосунок розгорнутий та доступний онлайн.

## **3. СТРАТЕГІЇ ПРОЕКТУВАННЯ РІШЕННЯ**
### **3.1. Стратегія архітектурного проектування**

Застосунок реалізується як клієнт-серверна система з використанням SPA (Single Page Application) для клієнтської частини.

Архітектура системи:

Frontend (React SPA)  
↓  
REST API  
↓  
Backend (Node.js + Express)  
↓  
MongoDB

Основні модулі:

**Frontend**

- UI-модуль
- модуль управління нотатками
- Markdown-модуль
- модуль авторизації
- модуль експорту

**Backend**

- API-модуль
- модуль авторизації користувачів
- модуль управління нотатками
- модуль роботи з базою даних (MongoDB)

Взаємодія між клієнтом і сервером відбувається через REST API.
Дані нотаток зберігаються у базі даних MongoDB.

### **3.2. Стратегія технологічного проектування**

У проекті буде використано сучасний підхід до розробки SPA-застосунків із використанням бібліотеки React.

**Основні технології:**

**Frontend**

- React - для побудови компонентної архітектури інтерфейсу;
  
- JavaScript (ES6+) - основна мова програмування;
  
- Vite - для ініціалізації та збірки проекту;
  
- CSS Modules - для стилізації;
  
- react-markdown - для обробки Markdown;
  
- React Hooks - для управління станом та автозбереження;
  
- Axios - для взаємодії з сервером.

**Backend**

- Node.js - середовище виконання серверної частини;
  
- Express.js - для створення REST API;
 
- JWT (JSON Web Token) - для авторизації користувачів;
  
- bcrypt - для хешування паролів.

**База даних**

- MongoDB - для збереження нотаток та даних користувачів;
 
- Mongoose - для роботи з MongoDB через моделі.

**Інструменти**

- Git / GitHub - контроль версій;
  
- MongoDB Atlas - хмарна база даних;
  
- Vercel - для розгортання застосунку;
  
- VS Code - середовище розробки.
